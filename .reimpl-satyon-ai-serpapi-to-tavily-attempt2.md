# Re-implementation: Fix Review Issues for Unit satyon-ai-serpapi-to-tavily

The original migration plan is at: workspaces/SanFlash--SATYON-AI/.migration-plan-satyon-ai-serpapi-to-tavily.md

## Review Feedback

The following issues were found during code review. Fix these specific issues. Do NOT re-do the entire migration — only address the problems below.

### Issue 1 [MAJOR]: backend/query_classifier.py

**Problem:** The `tavily` source is not added to any entry in `CATEGORY_SOURCES`. Every category — including the `general` fallback — omits `tavily`. This means the query classifier will never auto-select Tavily for any query, so it will never run in parallel with other sources during normal operation. The migration plan explicitly flagged this: "The query_classifier.py source-selection logic may also need 'tavily' added to the general/web category if it gates which sources are activated automatically." Since `general` is the fallback used for all unclassified queries, the stated goal of parallel/simultaneous searching is not achieved for the default user experience.
**Suggestion:** Add `'tavily'` to the appropriate entries in `CATEGORY_SOURCES`. At minimum add it to `'general'` (the catch-all fallback) so it participates in standard web searches: `"general": ["google", "wikipedia", "stackoverflow", "github", "arxiv", "youtube", "kaggle", "tavily"]`. Consider also adding it to `'tutorial'` and `'dataset'` categories alongside `'google'` if broad web search is desired for those categories.

### Issue 2 [MINOR]: backend/requirements.txt

**Problem:** `tavily-python` is added as a dependency but is never imported or used anywhere in the codebase. The implementation uses `aiohttp` to call the Tavily REST API directly (matching the `google_api.py` pattern), making the SDK an unused transitive dependency that bloats the install and adds a supply-chain surface area.
**Suggestion:** Remove `tavily-python` from `backend/requirements.txt` and `requirements.txt` since the implementation uses `aiohttp` directly. If you later decide to use the official SDK, add it back at that point.

### Issue 3 [MINOR]: requirements.txt

**Problem:** Same unused `tavily-python` dependency as in `backend/requirements.txt`.
**Suggestion:** Remove `tavily-python` from the root `requirements.txt` for the same reason as above.

### Issue 4 [MINOR]: backend/api_clients/tavily_api.py

**Problem:** The demo fallback results in `_get_demo_results()` use `https://www.google.com/search?q=...` as the URL while labeling the source as `'tavily'`. This is misleading — if a user clicks a demo result, they land on Google, not Tavily.
**Suggestion:** Use `https://tavily.com` or a neutral placeholder URL (e.g. `https://example.com`) for demo results, or note in the snippet that these are demo/placeholder results.

### Issue 5 [MINOR]: .migration-plan-satyon-ai-serpapi-to-tavily.md

**Problem:** A planning/migration artifact file has been committed to the repository root. Planning documents are not part of the application code and clutter the repo history.
**Suggestion:** Remove this file from the commit (add it to `.gitignore` or simply delete it). Migration plans belong in PR descriptions, wikis, or issue trackers — not committed source files.
